# 基于阿里云微服务生态的轻量级电商系统开发方案

## 1. 项目概述

本项目面向毕业设计场景，建设一套基于阿里云微服务生态的轻量级电商系统。系统采用前后端分离架构，覆盖商城用户端、管理后台和后端微服务集群三部分，围绕用户、商品、购物车、订单、支付、通知、搜索推荐等核心业务场景实现完整业务闭环。

方案设计坚持“满足任务书要求、保证系统可运行、控制实现复杂度”三项原则，在技术体系上以 `Spring Boot + Spring Cloud Alibaba + Nacos + Sentinel + Seata/RocketMQ` 为主，同时避免过度拆分和复杂治理，确保系统具备较强的工程落地性、可演示性和可部署性。

## 2. 建设目标

本系统的开发目标如下：

- 实现商城用户端、管理后台、后端微服务三层分离架构。
- 完成用户认证、商品管理、购物车、订单、支付、通知公告、搜索推荐等核心功能。
- 实现服务注册与发现、统一配置管理、网关路由、限流熔断、服务调用、分布式事务、异步消息等微服务治理能力。
- 支持 Docker 容器化部署和一键启动，满足毕业设计交付要求。

## 3. 总体设计原则

### 3.1 设计原则

- 以业务闭环优先，先保证“可浏览、可下单、可支付、可管理、可部署”。
- 以中等粒度微服务拆分为主，避免服务过多导致联调复杂。
- 以主流技术方案为主，优先选用与毕业设计主题契合度高、资料成熟、实现路径清晰的组件。
- 以轻量实现为准，复杂算法和重型基础设施仅做必要保留，不追求工业级超大规模能力。

### 3.2 架构思路

系统采用前后端分离 + 微服务架构，客户端统一通过 API 网关访问后端服务，后端服务通过注册中心完成服务发现，通过配置中心完成统一配置管理，通过 Sentinel 提供流量治理能力，通过 Seata 或 RocketMQ 实现关键业务链路的一致性与解耦。

| 客户端 | 选型 |

### 4.3 选型说明

- `Spring Cloud Alibaba` 与题目“阿里云微服务生态”高度匹配，适合作为系统主技术路线。
- `Nacos` 同时承担注册中心与配置中心职责，可降低基础设施复杂度。
- `Sentinel` 适合展示限流、熔断、降级等微服务治理能力。
- `Seata` 用于订单、库存、支付等关键场景的一致性控制。
- `RocketMQ` 用于异步通知、订单状态同步、服务解耦，便于体现事件驱动思想。
- 前端统一采用 `Vue 3` 技术栈，降低多端维护成本。

## 5. 系统总体架构

### 5.1 架构分层

系统整体分为以下三层：

1. 表现层  
   包括商城用户端和管理后台，分别服务普通用户和系统管理员。

2. 网关接入层  
   由 `gateway-service` 统一承接所有前端请求，负责路由转发、跨域处理、Token 校验、权限前置校验、限流熔断等。

3. 业务服务层  
   由多个微服务组成，分别承担认证用户、商品、交易、搜索推荐、通知公告等业务能力。

### 5.2 推荐微服务划分

本项目采用中等粒度微服务拆分，推荐拆分为以下 6 个核心服务：

- `gateway-service`：网关服务
- `auth-user-service`：认证与用户服务
- `product-service`：商品服务
- `trade-service`：交易服务
- `search-service`：搜索推荐服务
- `message-service`：通知公告服务

这种拆分方式既能体现微服务架构思想，也能控制开发复杂度，适合作为毕业设计落地方案。

## 6. 服务边界与模块职责

### 6.1 gateway-service

职责如下：

- 系统统一访问入口
- 根据路由规则转发到目标微服务
- 统一处理跨域请求
- 对 JWT 令牌进行解析与基础校验
- 对接口进行统一限流、熔断、降级
- 预留统一日志、链路透传、灰度扩展能力

### 6.2 auth-user-service

职责如下：

- 用户注册、登录、退出
- JWT 令牌签发与校验配合
- 用户信息维护
- 收货地址管理
- 角色与权限管理
- 管理员与普通用户身份区分

该服务统一承载认证与用户基础资料，减少服务数量，降低权限链路复杂度。

### 6.3 product-service

职责如下：

- 商品分类管理
- 商品 SPU/SKU 管理
- 商品图片、详情、属性维护
- 商品上下架管理
- 商品库存管理
- 后台商品运营管理

库存能力建议先内聚在商品服务中，不额外拆分独立库存服务，以避免订单、库存、商品之间调用过深。

### 6.4 trade-service

职责如下：

- 购物车管理
- 订单创建与订单查询
- 订单状态流转
- 支付单生成与支付状态更新
- 用户订单历史查询
- 管理后台订单管理

购物车与订单、支付关系紧密，因此统一归入交易服务，更利于简化调用链。

### 6.5 search-service

职责如下：

- 商品关键词搜索
- 条件筛选
- 结果排序
- 首页热门推荐
- 商品详情页同类推荐
- 基于用户行为的轻量个性化推荐

该服务强调“轻量可用”，优先实现基础搜索和规则推荐，不引入高复杂度算法。

### 6.6 message-service

职责如下：

- 公告发布
- 公告历史查询
- 站内消息生成
- 用户通知推送记录
- 消息已读状态管理

该服务用于承接后台公告和订单类异步通知，是前后端展示通知能力的重要支撑。

## 8. 核心业务流程设计

### 8.1 用户认证流程

1. 用户在商城端或管理后台提交账号密码。
2. 请求经网关转发至认证用户服务。
3. 认证成功后生成 JWT 并返回前端。
4. 前端后续请求在请求头中携带 Token。
5. 网关对 Token 进行校验后放行到目标服务。
6. 目标服务根据用户角色判断接口访问权限。

### 8.2 商品浏览与搜索流程

1. 用户进入首页或商品列表页。
2. 普通商品展示请求由商品服务直接返回。
3. 搜索、筛选、排序等请求由搜索服务处理。
4. 搜索服务从索引数据中返回结果，降低对商品主业务表的直接压力。
5. 用户进入商品详情页后，系统同时返回相关推荐结果。

### 8.3 加入购物车流程

1. 用户在商品详情页选择 SKU 和数量。
2. 前端调用交易服务购物车接口。
3. 交易服务校验商品是否上架、库存是否充足。
4. 校验通过后将商品写入购物车。
5. 前端可在购物车页面查询购物车明细并调整数量。

### 8.4 订单创建流程

1. 用户从购物车选择商品并提交订单。
2. 交易服务校验商品价格、库存、购买数量。
3. 创建订单主表和订单明细表记录。
4. 调用库存扣减逻辑，减少可售库存。
5. 生成支付记录，订单状态置为待支付。
6. 成功后返回订单编号和支付信息。

订单、库存、支付数据一致性可优先采用 Seata 控制；如后续开发中需要进一步简化，也可调整为“本地事务 + RocketMQ 最终一致性”的实现方式。

### 8.5 支付流程

1. 用户点击支付按钮发起支付请求。
2. 交易服务生成支付单或更新支付状态。
3. 系统采用模拟支付方式完成支付成功回调。
4. 支付成功后更新订单状态为已支付。
5. 同时发送订单支付成功事件到消息队列。
6. 消息服务消费事件并生成站内通知。

本项目建议采用模拟支付，既保留支付服务设计，又避免真实第三方支付接入带来的复杂度。

### 8.6 通知公告流程

1. 管理员在后台发布公告。
2. 公告写入通知公告服务数据库。
3. 普通用户可在用户端查询公告列表和详情。
4. 订单创建、支付成功、取消订单等业务事件通过 RocketMQ 投递。
5. 消息服务消费事件后生成用户通知记录。

## 9. 数据模型设计

### 9.1 数据库划分原则

建议采用“每个微服务对应一个独立业务库”的方式进行逻辑拆分，以体现微服务边界。实际部署时可先放在同一个 MySQL 实例中，通过不同数据库或不同 schema 区分。

### 9.2 认证与用户库

数据库：`auth_user_db`

主要表设计：

- `user`：用户基础信息
- `role`：角色表
- `permission`：权限表
- `user_role`：用户与角色关联表
- `user_address`：用户收货地址表

### 9.3 商品库

数据库：`product_db`

主要表设计：

- `category`：商品分类表
- `product_spu`：商品主表
- `product_sku`：商品规格表
- `product_stock`：库存表
- `product_image`：商品图片表

### 9.4 交易库

数据库：`trade_db`

主要表设计：

- `cart`：购物车主表
- `cart_item`：购物车明细表
- `order_info`：订单主表
- `order_item`：订单明细表
- `payment_info`：支付记录表

### 9.5 搜索推荐库

数据库：`search_db`

主要表设计：

- `product_search_index`：商品搜索索引表
- `user_behavior`：用户行为表

### 9.6 通知公告库

数据库：`message_db`

主要表设计：

- `notice`：公告表
- `user_message`：用户消息表
- `message_read_log`：消息已读记录表

### 9.7 公共字段设计建议

为便于统一管理，主要业务表建议统一包含以下字段：

- `id`
- `create_time`
- `update_time`
- `create_by`
- `update_by`
- `deleted`
- `remark`

### 9.8 核心状态设计建议

订单状态建议包含：

- `PENDING_PAYMENT`
- `PAID`
- `CANCELLED`
- `SHIPPED`
- `COMPLETED`

支付状态建议包含：

- `UNPAID`
- `PAYING`
- `PAID`
- `FAILED`

商品状态建议包含：

- `DRAFT`
- `ON_SALE`
- `OFF_SALE`

## 10. 接口设计规范

### 10.1 设计原则

- 接口风格采用 RESTful API
- 统一返回结构
- 前后端使用 JSON 进行数据交换
- 接口按“公共接口、用户接口、管理接口”进行分层设计

### 10.2 统一返回结构

建议统一返回如下结构：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

### 10.3 接口分类

公共接口包括：

- 用户注册
- 用户登录
- 商品列表查询
- 商品详情查询
- 商品搜索

用户接口包括：

- 收货地址管理
- 购物车管理
- 订单提交与查询
- 支付操作
- 个人通知查询

管理接口包括：

- 商品新增、编辑、删除、上下架
- 分类管理
- 库存调整
- 用户管理
- 订单管理
- 公告管理

### 10.4 通用参数规范

建议统一采用以下参数命名：

- 分页：`pageNum`、`pageSize`
- 排序：`sortField`、`sortOrder`
- 状态：`status`
- 时间区间：`startTime`、`endTime`

## 11. 搜索与推荐设计

### 11.1 搜索实现方案

为控制复杂度，搜索服务建议分阶段实现：

1. 第一阶段  
   基于 MySQL 条件查询实现关键词搜索、分类筛选、价格区间筛选、销量排序、价格排序等基础能力。

2. 第二阶段  
   如时间允许，可补充接入 Elasticsearch 优化搜索体验和查询性能。

### 11.2 推荐实现方案

推荐能力建议采用规则驱动的轻量实现：

- 首页热门商品推荐：按销量或浏览量排序
- 同类商品推荐：按商品分类返回相似商品
- 个性化推荐：根据用户最近浏览记录和购买记录进行简单规则推荐

该方案实现成本低、演示效果明确，适合毕业设计项目。

## 12. 微服务治理设计

### 12.1 服务注册与发现

所有微服务启动后注册到 Nacos，由 Nacos 统一维护服务实例信息。服务之间通过服务名进行调用，降低硬编码依赖。

### 12.2 配置中心

系统公共配置、数据库配置、缓存配置、服务开关等统一托管到 Nacos 配置中心，便于集中管理与动态调整。

### 12.3 网关路由

所有客户端请求统一进入网关，由网关根据路径规则转发到具体服务，同时在网关层做统一认证和公共拦截。

### 12.4 限流与熔断

基于 Sentinel 对以下场景进行保护：

- 登录接口防刷
- 商品搜索接口限流
- 订单提交接口保护
- 高并发商品详情访问保护

### 12.5 服务调用

服务间调用建议采用 OpenFeign，实现声明式远程调用，提升代码可读性和开发效率。

### 12.6 分布式事务

订单创建流程涉及订单表、库存表、支付表等多处数据变更，建议通过 Seata 实现关键链路事务一致性。

如后续实现中发现 Seata 集成成本偏高，可保留主方案不变，并在开发阶段对个别链路调整为 RocketMQ 异步最终一致性方案。

### 12.7 异步消息

基于 RocketMQ 实现以下异步场景：

- 订单创建后发送业务事件
- 支付成功后发送通知事件
- 公告发布后的消息分发
- 搜索索引更新触发

## 13. 安全与权限设计

### 13.1 认证机制

系统采用 JWT 作为统一认证凭证。用户或管理员登录成功后获取 Token，后续请求通过请求头传递给网关和业务服务进行校验。

### 13.2 授权机制

采用基于角色的权限控制模型：

- 普通用户：商城浏览、购物、订单、个人信息管理
- 管理员：商品管理、订单管理、公告管理、用户管理

### 13.3 安全控制建议

- 登录密码加密存储
- 敏感接口进行权限校验
- 关键操作记录日志
- 防止越权访问用户订单和个人信息

## 14. 测试方案

为满足任务书要求，系统测试建议分为以下几类：

- 单元测试：针对核心业务方法和工具类
- 接口测试：针对登录、商品、购物车、订单、支付、公告等核心接口
- 集成测试：针对服务调用链和事务链路
- 性能测试：针对商品查询、搜索、订单提交等关键接口

测试重点应覆盖以下业务闭环：

- 用户注册登录闭环
- 商品浏览搜索闭环
- 购物车与下单闭环
- 支付与订单状态更新闭环
- 公告发布与通知查询闭环

## 15. 部署方案

### 15.1 部署目标

系统部署以“本地可运行、容器化可启动、一键化可演示”为目标，推荐采用 Docker Compose 完成整体编排。

### 15.2 基础设施容器

建议部署以下基础组件：

- `mysql`
- `redis`
- `nacos`
- `sentinel-dashboard`
- `seata-server`
- `rocketmq-namesrv`
- `rocketmq-broker`

### 15.3 业务服务容器

建议部署以下业务服务：

- `gateway-service`
- `auth-user-service`
- `product-service`
- `trade-service`
- `search-service`
- `message-service`

### 15.4 前端应用容器

建议部署以下前端应用：

- `shop-web`
- `admin-web`

### 15.5 部署拓扑说明

部署时，前端应用通过网关访问所有业务服务；业务服务通过 Nacos 进行注册发现，通过 MySQL、Redis、RocketMQ 等中间件完成业务支撑。所有容器由 Docker Compose 统一编排，便于开发环境和答辩演示环境快速启动。

## 16. 开发落地建议

为保证开发过程清晰、风险可控，建议后续实现按以下顺序推进：

1. 搭建基础工程骨架，包括父工程、公共模块、网关、Nacos 接入、配置中心接入。
2. 完成认证与用户服务，实现注册、登录、权限、收货地址。
3. 完成商品服务，实现分类、商品、库存、上下架。
4. 完成交易服务，实现购物车、订单、支付模拟。
5. 完成通知公告服务，实现后台公告和用户通知。
6. 完成搜索推荐服务，实现搜索筛选和轻量推荐。
7. 完成前端商城用户端与管理后台页面对接。
8. 完成 Docker 化、一键部署、测试补充和文档整理。

## 17. 最终方案结论

本项目最终采用“前后端分离 + 中等粒度微服务”的整体建设方案。后端基于 `Spring Boot + Spring Cloud Alibaba` 构建，围绕网关、认证用户、商品、交易、搜索推荐、通知公告 6 个核心服务开展设计；前端包括商城用户端与管理后台两套应用；基础设施采用 `Nacos + Sentinel + Seata + RocketMQ + MySQL + Redis` 组合完成服务治理和业务支撑。

该方案能够较好满足毕业设计任务书中对微服务架构、电商业务场景、服务治理能力、容器化部署和系统交付的要求，同时保持适中的实现复杂度，适合作为后续完整开发的正式蓝本。
